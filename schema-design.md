# Smart Clinic - Database Schema Design

This document outlines the hybrid database architecture for the Smart Clinic System. We use MySQL for structured, relational data and MongoDB for flexible, unstructured document data.

## 1. MySQL Database Design (Relational Data)
This relational database handles our core entities. Strict schemas ensure data integrity across users and schedules.

### Table: admin
Stores portal administrators who manage the system.
* **id**: INT, Primary Key, Auto Increment
* **username**: VARCHAR(50), UNIQUE, NOT NULL
* **password**: VARCHAR(255), NOT NULL
* **email**: VARCHAR(100), UNIQUE, NOT NULL

### Table: doctors
Stores information about the medical staff.
* **id**: INT, Primary Key, Auto Increment
* **name**: VARCHAR(100), NOT NULL
* **specialization**: VARCHAR(100), NOT NULL
* **phone**: VARCHAR(15), NOT NULL
* **email**: VARCHAR(100), UNIQUE, NOT NULL
* **is_available**: BOOLEAN, DEFAULT TRUE

### Table: patients
Stores the registered patients who use the portal.
* **id**: INT, Primary Key, Auto Increment
* **name**: VARCHAR(100), NOT NULL
* **dob**: DATE, NOT NULL
* **email**: VARCHAR(100), UNIQUE, NOT NULL
* **phone**: VARCHAR(15), NOT NULL

### Table: appointments
Connects patients and doctors for scheduled visits.
* **id**: INT, Primary Key, Auto Increment
* **patient_id**: INT, Foreign Key (References patients.id), NOT NULL
* **doctor_id**: INT, Foreign Key (References doctors.id), NOT NULL
* **appointment_date**: DATETIME, NOT NULL
* **status**: VARCHAR(20), NOT NULL (e.g., 'Scheduled', 'Completed', 'Cancelled')

## 2. MongoDB Collection Design (Document Data)
We use MongoDB for prescriptions because medical records often require flexible, nested data structures (like lists of medications) that do not fit neatly into rigid SQL tables.
